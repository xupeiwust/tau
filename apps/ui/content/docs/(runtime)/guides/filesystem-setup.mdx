# Filesystem Setup

Configure the filesystem backend that kernels use to read project files. The framework provides three built-in constructors: `fromNodeFS` for disk access, `fromMemoryFS` for in-memory content, and `fromFsLike` for any fs-compatible object (ZenFS, BrowserFS, memfs, polyfills).

## Prerequisites

- [Install @taucad/runtime](/docs/getting-started/installation)
- Completed the [Quick Start](/docs/getting-started/quick-start)

## Goal

Choose and configure a `RuntimeFileSystem` implementation so your runtime client can read (and optionally write) project files during render, export, and parameter extraction.

## Steps

### 1. Import Filesystem Constructors from @taucad/runtime

All three constructors are exported from the main package:

```typescript
import { fromNodeFS, fromMemoryFS, fromFsLike } from '@taucad/runtime';
```

### 2. Use fromNodeFS for Disk-Based Projects

Use `fromNodeFS` when your project files live on disk. Pass the base path; all file operations are resolved relative to it. Requires Node.js (not available in browser builds).

```typescript
import { createRuntimeClient, fromNodeFS } from '@taucad/runtime';
import { replicad } from '@taucad/runtime/kernels';
import { esbuild } from '@taucad/runtime/bundler';

const client = createRuntimeClient({
  kernels: [replicad()],
  bundlers: [esbuild()],
  fileSystem: fromNodeFS('/path/to/project'),
});
```

Use this when running in Node.js (CLI, tests, server-side) and when you need persistent cache (e.g., geometry cache, parameter cache).

### 3. Use fromMemoryFS for In-Memory Content

Use `fromMemoryFS` when file content is passed directly (e.g., from an editor) or when you want no disk I/O. Optionally seed with initial files:

```typescript
import { createRuntimeClient, fromMemoryFS } from '@taucad/runtime';
import { replicad } from '@taucad/runtime/kernels';
import { esbuild } from '@taucad/runtime/bundler';

const filesystem = fromMemoryFS({
  'main.ts': `
    import { drawRoundedRectangle } from 'replicad';
    export default function main() {
      return drawRoundedRectangle(30, 50, 5).sketchOnPlane('XY').extrude(10);
    }
  `,
  'lib/utils.ts': 'export function helper() { return 42; }',
});

const client = createRuntimeClient({
  kernels: [replicad()],
  bundlers: [esbuild()],
  fileSystem: filesystem,
});
```

When you call `client.render({ file: 'main.ts', code: '...' })`, the `code` is written to the in-memory store before the kernel runs. Use `fromMemoryFS()` with no args for an empty store that you populate via `render`/`connect`.

### 4. Use fromFsLike for Any fs-Compatible Object

Use `fromFsLike` when you have an fs-compatible object with a `promises` namespace (e.g., ZenFS, BrowserFS, memfs, or any polyfill). Pass the fs instance:

```typescript
import { createRuntimeClient, fromFsLike } from '@taucad/runtime';
import { fs } from '@zenfs/core';

const filesystem = fromFsLike(fs, '/');

const client = createRuntimeClient({
  kernels: [replicad()],
  bundlers: [esbuild()],
  fileSystem: filesystem,
});
```

Use this when you have a `node:fs`-compatible object in any environment -- browser virtual filesystems (ZenFS IndexedDB, BrowserFS), in-memory polyfills (memfs), or any library that provides `fs.promises`.

### 5. Understand the RuntimeFileSystem Interface

Filesystem backends implement `RuntimeFileSystemBase` -- 11 primitives matching `fs.promises` semantics:

| Method      | Signature                                            | Purpose                                                 |
| ----------- | ---------------------------------------------------- | ------------------------------------------------------- |
| `readFile`  | `(path, encoding?) => Promise<string \| Uint8Array>` | Read file as text (`'utf8'`) or binary                  |
| `writeFile` | `(path, data) => Promise<void>`                      | Write text or binary                                    |
| `mkdir`     | `(path, options?) => Promise<void>`                  | Create directory (supports `{ recursive: true }`)       |
| `readdir`   | `(path) => Promise<string[]>`                        | List directory entries                                  |
| `unlink`    | `(path) => Promise<void>`                            | Delete file                                             |
| `rmdir`     | `(path) => Promise<void>`                            | Remove directory                                        |
| `rename`    | `(oldPath, newPath) => Promise<void>`                | Rename/move file or directory                           |
| `stat`      | `(path) => Promise<FileStat>`                        | Get file/dir metadata (`FileStat` from `@taucad/types`) |
| `lstat`     | `(path) => Promise<FileStat>`                        | Like stat, but does not follow symlinks                 |
| `exists`    | `(path) => Promise<boolean>`                         | Check if path exists                                    |

The framework wraps the base with `createRuntimeFileSystem(base)` to add enhanced helper methods with default implementations:

| Method            | Signature                                          | Default Implementation                       |
| ----------------- | -------------------------------------------------- | -------------------------------------------- |
| `readFiles`       | `(paths) => Promise<Record<string, Uint8Array>>`   | `Promise.all(paths.map(readFile))`           |
| `readdirContents` | `(dirPath) => Promise<Record<string, Uint8Array>>` | `readdir` + `stat` + `readFile` (files only) |
| `readdirStat`     | `(dirPath) => Promise<FileStatEntry[]>`            | `readdir` + `stat` per entry                 |
| `ensureDir`       | `(path) => Promise<void>`                          | `mkdir(path, { recursive: true })`           |

All paths are absolute. Backends can supply optimized overrides for any enhanced method (e.g., batch reads at the storage layer).

## When to Use Each

| Constructor                 | Use when                                                      |
| --------------------------- | ------------------------------------------------------------- |
| `fromNodeFS(basePath)`      | Node.js environment, disk-based projects, persistent cache    |
| `fromMemoryFS(files?)`      | Editor-driven content, tests, no disk I/O, ephemeral cache    |
| `fromFsLike(fs, rootPath?)` | Any fs-compatible object (ZenFS, BrowserFS, memfs, polyfills) |

## Variations

- **Reading files from disk vs in-memory**: With `fromNodeFS`, kernels read from the real filesystem. With `fromMemoryFS`, they read from the in-memory Map. The bundler and kernels use the same interface; only the backend differs.
- **Custom implementation**: Implement `RuntimeFileSystem` directly for custom backends (e.g., cloud storage, virtual mounts). See [API Reference: Types](/docs/api/types) for the full interface.
- **Filesystem bridge**: For advanced setups (e.g., UI exposing a filesystem to a worker), use `createBridgePort` and `exposeFileSystem` from `@taucad/runtime/filesystem`. Both `createBridgePort` and `createFileSystemBridge` return a `BridgeHandle` with `{ port, dispose() }` -- call `dispose()` when the bridge is no longer needed to release the underlying MessagePort resources.
