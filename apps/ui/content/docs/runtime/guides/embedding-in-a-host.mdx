# Embedding in a Host

The `transport` option on `createRuntimeClient` is the single seam where the host hands filesystem and wire primitives to the runtime. Every wire concern — filesystem, file pool SAB, abort signalling — is closed over by the transport at construction. `client.connect()` takes no arguments in v6; it only runs the handshake for the transport that was supplied at construction.

This guide covers the three scenarios where explicit transport wiring is the right choice:

1. **Worker-hosted filesystems** -- the host owns the filesystem worker and exposes it to the runtime via `fromChannelFs(worker)`.
2. **Externally allocated `filePoolBuffer` SABs** -- the host pre-allocates a `SharedArrayBuffer` for zero-copy file content sharing across workers.
3. **Deferred client creation** -- the runtime client is constructed once the host knows which filesystem to bind to.

If none of these apply, head back to the [Quick Start](../getting-started/quick-start) or [Live Rendering](./live-rendering) guides -- they are the canonical npm-consumer paths.

## Why a Host Would Wire This Manually

A host (CAD editor, IDE, multi-user platform) typically owns the filesystem and worker explicitly:

- A single shared filesystem worker serving multiple runtime clients (one per chat panel, one per render preview)
- Externally managed lifecycle (the host already owns "the project filesystem"; the runtime is just a consumer)
- Cross-worker zero-copy file content via `SharedArrayBuffer`-backed `FilePool`
- The ability to terminate and re-create runtime clients without losing the underlying filesystem

`createRuntimeClient({ ..., transport: webWorkerTransport({ fileSystem, filePoolBuffer }) })` is the seam where the host hands these primitives to the runtime.

## Opaque `RuntimeFileSystem` values

The **`fileSystem`** field on `webWorkerTransport({ ... })`, `inProcessTransport({ ... })`, and friends accepts an opaque [`RuntimeFileSystem`](../api/filesystem) value. The value is produced by one of the bundled factory functions — consumers cannot inspect its internals. The transport constructs the underlying bridge primitives from it:

| Factory                              | When to use                                  |
| ------------------------------------ | -------------------------------------------- |
| `fromMemoryFs({ 'main.ts': '...' })` | Static fixtures, tests, CLI one-shots        |
| `fromFsLike(myFs)`                   | Wrap a custom `RuntimeFileSystemBase` impl   |
| `fromNodeFs('/projects/widget')`     | Headless / Node.js (CLI, benchmarks, server) |
| `fromBrowserFs(directoryHandle)`     | Browser File System Access API root          |
| `fromChannelFs(fileManagerWorker)`   | Editor host with a dedicated FS worker       |

### Worker-hosted filesystems

When the host owns a File Manager Worker (the canonical editor pattern), wrap it with `fromChannelFs(...)` and pass the handle to `webWorkerTransport`:

```typescript
import { createRuntimeClient, presets } from '@taucad/runtime';
import { webWorkerTransport } from '@taucad/runtime/transport/web';
import { fromChannelFs } from '@taucad/runtime/filesystem';

const fileManagerWorker = new Worker(new URL('./fm-worker.ts', import.meta.url));

const client = createRuntimeClient({
  ...presets.all(),
  transport: webWorkerTransport({
    fileSystem: fromChannelFs(fileManagerWorker),
  }),
});
await client.connect();

const result = await client.export('glb', { file: 'main.ts' });
if (!result.success) throw new Error(`Export failed: ${result.issues[0]?.message}`);

client.terminate();
```

The transport defaults to its bundled worker entry — pass `url:` only when hosting a custom worker module.

The transport constructs the FS bridge `MessagePort` internally and disposes it on `client.terminate()`. The host never sees the port and never needs to call `dispose()` itself.

### In-process filesystems

For tests, CLIs, and other same-thread scenarios, use `inProcessTransport` with an inline filesystem:

```typescript
import { createRuntimeClient, presets, fromMemoryFs } from '@taucad/runtime';
import { inProcessTransport } from '@taucad/runtime/transport/in-process';

const client = createRuntimeClient({
  ...presets.all(),
  transport: inProcessTransport({
    fileSystem: fromMemoryFs({
      'main.ts':
        "import { drawCircle } from 'replicad';\nexport default () => drawCircle(10).sketchOnPlane('XY').extrude(20);",
    }),
  }),
});
await client.connect();
```

The transport detects the inline arm and wires an in-process bridge that auto-propagates `watch` events when the wrapped FS exposes a `watch` method — so editor edits flow through the same dependency-invalidation pipeline as worker-hosted FSes.

## Externally Allocated `filePoolBuffer` SABs

Pair `fileSystem` with `filePoolBuffer` when the host has pre-allocated a `SharedArrayBuffer` for the runtime's file pool. Sharing the buffer across the FS worker, kernel worker, and runtime client gives you zero-copy reads of file contents -- the kernel reads bytes from shared memory rather than postMessage-ing them through the bridge.

`filePoolBuffer` is the **only** `SharedArrayBuffer` exposed on the public runtime API. Geometry-pool and cooperative-abort SABs are owned and allocated by the active [transport](../api/client) — never by the host. The host owns this one because the file pool's lifetime is bound to the file manager (not the runtime), and crossing that boundary requires the host to allocate the buffer.

```typescript
import { createRuntimeClient, presets } from '@taucad/runtime';
import { webWorkerTransport } from '@taucad/runtime/transport/web';
import { fromChannelFs } from '@taucad/runtime/filesystem';

const fileManagerWorker = new Worker(new URL('./fm-worker.ts', import.meta.url));
const filePoolBuffer = new SharedArrayBuffer(64 * 1024 * 1024);

const client = createRuntimeClient({
  ...presets.all(),
  transport: webWorkerTransport({
    fileSystem: fromChannelFs(fileManagerWorker),
    filePoolBuffer,
  }),
});
await client.connect();
```

The same `filePoolBuffer` option is supported on **wire** transports such as `webWorkerTransport` and `nodeWorkerTransport`. `inProcessTransport` does not accept a host `SharedArrayBuffer`; it can size the in-isolate file pool via the `files` option instead.

> Cross-origin isolation is required for any `SharedArrayBuffer` — both the host-allocated `filePoolBuffer` and the transport-owned geometry/abort SABs. See [Cross-Origin Isolation](./cross-origin-isolation) for the COOP/COEP headers your host must serve.

## Deferred Client Creation

When the host needs a runtime client before it knows which filesystem to bind to (e.g. the client is created at app boot and the project is chosen on user navigation), construct the client once the filesystem is known:

```typescript
import { createRuntimeClient, presets } from '@taucad/runtime';
import { inProcessTransport } from '@taucad/runtime/transport/in-process';
import { fromNodeFs } from '@taucad/runtime/filesystem/node';

// ...later, once the user picks a project:
const projectPath = '/Users/me/cad-projects/widget';
const client = createRuntimeClient({
  ...presets.all(),
  transport: inProcessTransport({ fileSystem: fromNodeFs(projectPath) }),
});
await client.connect();
```

Because every wire concern is closed over at construction time, the client stays in the `'unconnected'` lifecycle state until `connect()` is called — no worker is spawned, no transport handshake runs. This is the correct pattern for editors that want to instantiate clients lazily once the workspace is known.

## Lifecycle and Re-connection

`client.connect()` is **idempotent**: calling it a second time while already connected resolves immediately without re-running the handshake. To switch a client to a different project, terminate the existing client and create a fresh one with the new transport options.

```typescript
import { createRuntimeClient, presets, fromMemoryFs } from '@taucad/runtime';
import { inProcessTransport } from '@taucad/runtime/transport/in-process';

const client = createRuntimeClient({
  ...presets.all(),
  transport: inProcessTransport({
    fileSystem: fromMemoryFs({ 'main.ts': 'export default () => 1;' }),
  }),
});

// Idempotent: calling connect() again resolves immediately.
await client.connect();
await client.connect();

// To switch projects: terminate and create a new client.
client.terminate();
```

The lifecycle progression is `'unconnected' -> 'connecting' -> 'connected' -> 'terminated'`. Inspect the current state via `client.lifecycleState` -- helpful for host-level diagnostics and React state derivation.

### Deterministic `terminate()` and graceful `shutdown()`

`client.terminate()` is **synchronous** and observes seven invariants the host can rely on without timing-dependent shutdown choreography:

1. `client.lifecycleState` becomes `'terminated'` synchronously.
2. The transport's `close()` is invoked exactly once.
3. Every in-flight `connect()` / `openFile()` / `updateParameters()` / `setOptions()` / `export()` Promise rejects on the next microtask with `RuntimeTerminatedError` (`causeKind: 'explicit'`).
4. `client.on(...)` calls made after `terminate()` throw synchronously (subscription is no longer meaningful).
5. Subsequent commands reject synchronously with `RuntimeTerminatedError` (`causeKind: 'explicit'`).
6. A second `terminate()` call is a no-op (idempotent).
7. Captured connect options are cleared so a fresh client created from the same host primitives starts from a clean slate.

For orderly shutdown paths (`beforeunload`, server graceful-exit, integration test teardown) the asynchronous `shutdown({ drain? })` companion lets the host await in-flight intents before tearing down:

```typescript
function attachShutdown(client: { shutdown: (opts: { drain?: boolean }) => Promise<void> }): void {
  window.addEventListener('beforeunload', () => {
    void client.shutdown({ drain: true });
  });
}
```

`shutdown({ drain: true }){:ts}` snapshots the in-flight intents, awaits each via `Promise.allSettled{:ts}`, and then runs the same teardown as `terminate(){:ts}`. A concurrent `terminate(){:ts}` cancels the drain — pending intents reject with `RuntimeTerminatedError{:ts}` and the awaiting `shutdown(){:ts}` promise still resolves. Both methods are idempotent.

Wire this from your host after you construct the client — for example `attachShutdown(client){:ts}` where `client` is your `RuntimeClient{:ts}`.
