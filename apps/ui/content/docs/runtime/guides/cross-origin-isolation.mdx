# Cross-Origin Isolation

`@taucad/runtime` requires `crossOriginIsolated === true` for `SharedArrayBuffer`-backed features. SAB ownership is split across two layers and you should treat them as one isolation requirement:

- **Transport-owned SABs** â€” the active [transport](../api/client) allocates the geometry pool and the cooperative-abort channel internally via `RuntimeTransport.configureMemory`. The runtime client, worker client, and your application code never see these.
- **Host-owned SABs** â€” the multi-threaded OpenCASCADE WASM heap and the optional `filePool` you pass to `client.connect(...)` are allocated by the WASM kernel and the host respectively.

Either way, browsers only enter `crossOriginIsolated` state when the top-level document is served with the right combination of COOP, COEP, and CORP headers.

The runtime ships small adapters for the layers where headers are produced. Use them; do not duplicate header strings in your app.

## Why this matters

`SharedArrayBuffer` and other isolation-gated APIs are blocked by default to mitigate Spectre/Meltdown. To opt in, the document must declare:

| Header                         | Value          | Purpose                                                              |
| ------------------------------ | -------------- | -------------------------------------------------------------------- |
| `Cross-Origin-Opener-Policy`   | `same-origin`  | Isolates the top-level browsing context.                             |
| `Cross-Origin-Embedder-Policy` | `require-corp` | Forces every embedded resource to declare CORP.                      |
| `Cross-Origin-Resource-Policy` | `same-origin`  | Allows the document and same-origin subresources to load under COEP. |

Safari is strictly conformant: under `require-corp`, every subresource â€” including same-origin worker scripts and WASM binaries â€” must carry an explicit `Cross-Origin-Resource-Policy` header. Chromium and Firefox are more permissive, which is why a missing CORP header on a worker often shows up as a Safari-only "eternal loading" bug.

## Topology matrix

The runtime provides one adapter per response-producing layer. Pick the rows you ship:

| Layer                                | Adapter                                              | When to use                                       |
| ------------------------------------ | ---------------------------------------------------- | ------------------------------------------------- |
| Vite dev/preview server              | `@taucad/runtime/vite`                               | `pnpm vite` / `vite preview`.                     |
| React Router SSR (the HTML document) | `@taucad/runtime/react-router`                       | Any SSR host (Express, Netlify, Fly).             |
| Express/Connect production server    | `@taucad/runtime/cross-origin-isolation/express`     | `node server.ts` or any custom Express bootstrap. |
| Static host with declarative headers | Your host's config (e.g. `netlify.toml`, `_headers`) | CDN-fronted static deploys.                       |

Set the headers at every layer that produces a response. Missing any one of HTML, worker scripts, or WASM responses breaks Safari.

## Vite dev/preview

```typescript
import { crossOriginIsolation } from '@taucad/runtime/vite';
import { defineConfig } from 'vite';

export default defineConfig({
  plugins: [crossOriginIsolation()],
});
```

## Express production server

Mount `coiMiddleware()` **before** `express.static` so worker scripts and WASM responses carry CORP. The middleware also suppresses downstream `res.append()` of COI headers, so the same triple cannot be duplicated by a downstream `Response` writer (e.g. `@react-router/express`).

```typescript
import express, { type Express, type RequestHandler } from 'express';
import { createRequestHandler } from '@react-router/express';
import { coiMiddleware } from '@taucad/runtime/cross-origin-isolation/express';

type ServerBuild = Parameters<typeof createRequestHandler>[0]['build'];

export function createApp(build: ServerBuild): Express {
  const app = express();
  app.disable('x-powered-by');
  app.use(coiMiddleware() as RequestHandler);
  app.use('/assets', express.static('build/client/assets', { immutable: true, maxAge: '1y' }));
  app.use(express.static('build/client', { maxAge: '1h' }));
  app.all('*splat', createRequestHandler({ build }));
  return app;
}
```

## Static hosts

If you ship static assets behind a CDN, configure the headers declaratively. For Netlify:

```toml
[[headers]]
  for = "/*"
  [headers.values]
    Cross-Origin-Opener-Policy = "same-origin"
    Cross-Origin-Embedder-Policy = "require-corp"
    Cross-Origin-Resource-Policy = "same-origin"
```

## Verifying

After deploying or starting the server, verify every response class. For each, all three headers must appear exactly once:

```bash
# HTML document
curl -sI http://localhost:3000/ | grep -i cross-origin

# Worker script (any *.worker-*.js asset)
curl -sI http://localhost:3000/assets/file-manager.worker-XXXXX.js | grep -i cross-origin

# WASM binary
curl -sI http://localhost:3000/draco_decoder_gltf.wasm | grep -i cross-origin
```

In the browser, open DevTools and confirm `crossOriginIsolated === true`:

```javascript
console.log(crossOriginIsolated);
console.log(typeof SharedArrayBuffer);
```

If isolation is off, `RuntimeClient` emits a structured warning at boot via its log stream so the failure is observable in production telemetry, not silently swallowed.

## Cross-origin APIs

When the isolated document fetches a different origin (e.g. an analytics or asset CDN), that origin must opt in with `Cross-Origin-Resource-Policy: cross-origin`. Use `@taucad/runtime/cross-origin-isolation`'s `apiHeaders` constant or `applyApiHeaders()` to set this on your API responses.
