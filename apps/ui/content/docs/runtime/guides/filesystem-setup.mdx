# Set Up the Filesystem

Configure the filesystem backend that kernels use to read project files. The framework provides three built-in constructors: `fromNodeFs` for disk access, `fromMemoryFs` for in-memory content, and `fromFsLike` for any fs-compatible object (BrowserFS, memfs, polyfills).

## Prerequisites

- [Install @taucad/runtime](../getting-started/installation)
- Completed the [Quick Start](../getting-started/quick-start)

## Goal

Choose and configure a filesystem implementation so your runtime client can read (and optionally write) project files during render, export, and parameter extraction.

## Steps

### 1. Import Filesystem Constructors

`fromMemoryFs` and `fromFsLike` are available from the main entry. `fromNodeFs` requires the Node.js subpath (it depends on Node.js APIs):

```typescript
import { fromMemoryFs, fromFsLike } from '@taucad/runtime';
import { fromNodeFs } from '@taucad/runtime/filesystem/node';
```

### 2. Use fromNodeFs for Disk-Based Projects

Use `fromNodeFs` when your project files live on disk. Pass the base path; all file operations are resolved relative to it. Requires Node.js (not available in browser projects).

```typescript
import { createRuntimeClient } from '@taucad/runtime';
import { fromNodeFs } from '@taucad/runtime/filesystem/node';
import { inProcessTransport } from '@taucad/runtime/transport/in-process';
import { replicad } from '@taucad/runtime/kernels';
import { esbuild } from '@taucad/runtime/bundler';

const client = createRuntimeClient({
  transport: inProcessTransport({ fileSystem: fromNodeFs('/path/to/project') }),
  kernels: [replicad()],
  bundlers: [esbuild()],
});
```

Use this when running in Node.js (CLI, tests, server-side) and when you need persistent cache (e.g., geometry cache, parameter cache).

### 3. Use fromMemoryFs for In-Memory Content

Use `fromMemoryFs` when file content is passed directly (e.g., from an editor) or when you want no disk I/O. Optionally seed with initial files:

```typescript
import { createRuntimeClient, fromMemoryFs } from '@taucad/runtime';
import { inProcessTransport } from '@taucad/runtime/transport/in-process';
import { replicad } from '@taucad/runtime/kernels';
import { esbuild } from '@taucad/runtime/bundler';

const filesystem = fromMemoryFs({
  'main.ts': `
    import { drawRoundedRectangle } from 'replicad';
    export default function main() {
      return drawRoundedRectangle(30, 50, 5).sketchOnPlane('XY').extrude(10);
    }
  `,
  'lib/utils.ts': 'export function helper() { return 42; }',
});

const client = createRuntimeClient({
  transport: inProcessTransport({ fileSystem: filesystem }),
  kernels: [replicad()],
  bundlers: [esbuild()],
});
```

### 4. Use fromFsLike for Any fs-Compatible Object

Use `fromFsLike` when you have an fs-compatible object with a `promises` namespace (e.g., BrowserFS, memfs, or any polyfill). The `FsLike` interface requires `promises.readFile`, `promises.writeFile`, `promises.mkdir`, and the other standard `fs.promises` methods:

```typescript
import nodeFs from 'node:fs';
import { createRuntimeClient, fromFsLike } from '@taucad/runtime';
import { inProcessTransport } from '@taucad/runtime/transport/in-process';
import { replicad } from '@taucad/runtime/kernels';
import { esbuild } from '@taucad/runtime/bundler';

const filesystem = fromFsLike(nodeFs, '/path/to/project');

const client = createRuntimeClient({
  transport: inProcessTransport({ fileSystem: filesystem }),
  kernels: [replicad()],
  bundlers: [esbuild()],
});
```

### 5. Understand RuntimeFileSystemBase, RuntimeFileSystem, and RuntimeFileSystem (opaque)

The framework uses three related filesystem types:

**`RuntimeFileSystem`** -- The opaque connect-time value returned by `fromNodeFs`, `fromMemoryFs`, `fromFsLike`, `fromBrowserFs`, and `fromChannelFs`. Hand it to the wired transport callable at construction (`webWorkerTransport({ fileSystem })`, `inProcessTransport({ fileSystem })`, …); consumers cannot inspect its internals.

**`RuntimeFileSystemBase`** -- The 11-method contract bridged to the kernel. Matches `fs.promises` semantics: `readFile`, `writeFile`, `mkdir`, `readdir`, `unlink`, `rmdir`, `rename`, `stat`, `lstat`, `exists`.

**[`RuntimeFileSystem`](../api/filesystem)** (framework view) -- An enhanced interface that extends the base with convenience methods inside the worker. The framework wraps the base automatically via `createRuntimeFileSystem(base)`:

| Method            | Signature                                          | Default Implementation                       |
| ----------------- | -------------------------------------------------- | -------------------------------------------- |
| `readFiles`       | `(paths) => Promise<Record<string, Uint8Array>>`   | `Promise.all(paths.map(readFile))`           |
| `readdirContents` | `(dirPath) => Promise<Record<string, Uint8Array>>` | `readdir` + `stat` + `readFile` (files only) |
| `readdirStat`     | `(dirPath) => Promise<FileStatEntry[]>`            | `readdir` + `stat` per entry                 |
| `ensureDir`       | `(path) => Promise<void>`                          | `mkdir(path, { recursive: true })`           |

All paths are absolute. Backends can supply optimized overrides for any enhanced method (e.g., batch reads at the storage layer).

### 6. Use Filesystem Bridges for Cross-Worker Communication

When the filesystem owns a **`Worker`** (**`exposeFileSystem`** there) and the client runs in another isolate (**`createFileSystemBridge(worker)`** on the owning thread), the returned **`FileSystemBridge.port`** is already a **`Port`** suited for **`createBridgeProxy`** in that isolate — no duplicate public helper exists for forwarding; opaque **`fromChannelFs`** builds the transferable **`MessagePort`** internally (**`packages/runtime`** `transport/_internal`).

For the inverse pattern — you host handler objects in-process and hand a port to a worker that will call you — use **`createBridgePort(handlers)`**, which returns **`BridgePort`** (`raw` **`MessagePort`**) for **`postMessage(..., [port])`**.

```typescript
import { createBridgePort } from '@taucad/runtime/transport-internals';
import { fromNodeFs } from '@taucad/runtime/filesystem/node';

const fileSystem = fromNodeFs('/path/to/project');
const { port, dispose } = createBridgePort(fileSystem);

const worker = new Worker(new URL('./worker.ts', import.meta.url), { type: 'module' });
worker.postMessage({ port }, [port]);

const cleanup = () => {
  worker.terminate();
  dispose();
};
```

**`createFileSystemBridge(worker, options?)`** -- Creates a **`MessageChannel`**, posts the server end to the FS worker, returns **`FileSystemBridge`** with a **`wrapMessagePort`**-backed **`port`** for same-isolate **`createBridgeProxy`**.

**`exposeFileSystem(handlers, options?)`** -- Worker-side counterpart; listens for the connect envelope (see **`filesystemBridgeConnectMessageType`** in **`@taucad/runtime/transport-internals`**).

Call `dispose()` on either handle when the bridge is no longer needed.

## When to Use Each

| Constructor                 | Use when                                                   |
| --------------------------- | ---------------------------------------------------------- |
| `fromNodeFs(basePath)`      | Node.js environment, disk-based projects, persistent cache |
| `fromMemoryFs(files?)`      | Editor-driven content, tests, no disk I/O, ephemeral cache |
| `fromFsLike(fs, rootPath?)` | Any fs-compatible object (BrowserFS, memfs, polyfills)     |

## Variations

- **Custom implementation**: Implement `RuntimeFileSystemBase` directly for custom backends (e.g., cloud storage, virtual mounts). See [API Reference: Filesystem](../api/filesystem) for the full interface.
- **Filesystem bridge**: For custom worker topologies, use **`createBridgePort`** and **`createFileSystemBridge`** from **`@taucad/runtime/transport-internals`**. For opaque transport wiring, prefer **`fromChannelFs(worker)`** on the main package.
