This guide explains how to add a new CAD kernel to Tau. It's based on the JSCAD kernel integration and serves as a template for future kernels.

## Overview

Tau's multi-kernel architecture allows different CAD engines to be plugged in as Web Workers. Each kernel follows the same interface pattern, making it straightforward to add new ones.

### What You'll Create

1. **Worker** — Runs in a Web Worker, handles geometry computation
2. **Kernel Registry Entry** — Configuration in the types library
3. **Worker Registration** — Add worker URL and options to the kernel config
4. **Examples** (optional) — Sample builds to showcase the kernel
5. **AI Configuration** (optional) — Prompts for AI-assisted modeling

## Step 1: Create the Worker

Create your worker at:

```
apps/ui/app/components/geometry/kernel/<kernel>/<kernel>.worker.ts
```

### Extend KernelWorker

Import and extend the base `KernelWorker` class:

```typescript
import { expose } from 'comlink';
import type { ExportFormat } from '@taucad/types';
import type { CreateGeometryResult, ExportGeometryResult, GetParametersResult, Tessellation } from '@taucad/kernels';
import { createKernelSuccess, createKernelError } from '@taucad/kernels';
import { KernelWorker } from '#components/geometry/kernel/utils/kernel-worker.js';

class MyKernelWorker extends KernelWorker {
  protected readonly name = 'MyKernelWorker';

  // Define supported export formats
  protected static override readonly supportedExportFormats: ExportFormat[] = ['stl', 'glb'];

  // Implement required abstract methods...
}

// Expose via Comlink
const service = new MyKernelWorker();
expose(service);
```

### Implement Required Methods

#### `onCanHandle(filename, extension)`

Quick check to determine if this worker can process the file. **Must be fast** — don't initialize heavy runtimes here.

```typescript
protected async onCanHandle(filename: string, extension: string): Promise<boolean> {
  // Check file extension
  if (extension !== 'myext') {
    return false;
  }

  // Optionally check file content for specific patterns
  const code = await this.readFile(filename, 'utf8');
  return code.includes('mykernel-signature');
}
```

#### `onGetParameters(path)`

Parse the file and extract parameters with their JSON Schema.

```typescript
protected async onGetParameters(path: string): Promise<GetParametersResult> {
  try {
    const code = await this.readFile(path, 'utf8');
    const { defaultParameters, jsonSchema } = parseParameters(code);

    return createKernelSuccess({
      defaultParameters,
      jsonSchema,
    });
  } catch (error) {
    return createKernelError({
      message: `Failed to extract parameters: ${error}`,
      type: 'code',
    });
  }
}
```

#### `onCreateGeometry(path, parameters, geometryId?)`

Execute the code and generate geometry (typically as glTF).

```typescript
protected async onCreateGeometry(
  path: string,
  parameters: Record<string, unknown>,
): Promise<CreateGeometryResult> {
  try {
    const code = await this.readFile(path, 'utf8');
    const geometry = await executeCode(code, parameters);
    const gltfBlob = await convertToGltf(geometry);

    return createKernelSuccess({
      geometries: [{
        id: geometryId ?? 'default',
        type: 'model/gltf-binary',
        blob: gltfBlob,
      }],
    });
  } catch (error) {
    return createKernelError({
      message: `Geometry computation failed: ${error}`,
      type: 'kernel',
    });
  }
}
```

#### `onExportGeometry(fileType, tessellation?)`

Export geometry to the requested format.

```typescript
protected async onExportGeometry(
  fileType: ExportFormat,
  tessellation?: Tessellation,
): Promise<ExportGeometryResult> {
  // Export from cached geometry
  const blob = await this.exportToFormat(fileType);

  return createKernelSuccess({
    files: [{ name: `model.${fileType}`, blob }],
  });
}
```

## Step 2: Register the Worker

The kernel machine dynamically creates workers from a `KernelModules` array injected at startup. Each entry bundles an `id`, `url`, and optional `options`. Array position defines `canHandle` priority (first match wins).

### First-party kernels

For kernels shipped with Tau, add an entry to `defaultKernelModules` in `apps/ui/app/constants/kernel-workers.ts`:

```typescript
import myKernelWorkerUrl from '#components/geometry/kernel/mykernel/mykernel.worker.js?url';

export const defaultKernelModules: KernelModules = [
  // ... existing entries (order = canHandle priority)
  { id: 'mykernel', url: myKernelWorkerUrl, options: { /* init options */ } },
  { id: 'tau', url: tauWorkerUrl }, // tau is typically last
];
```

### Third-party kernels

External consumers can extend the default config without modifying Tau's source:

```typescript
import { defaultKernelModules } from '#constants/kernel-workers.js';
import type { KernelModules } from '@taucad/kernels';

const extendedModules: KernelModules = [
  ...defaultKernelModules,
  { id: 'manifold', url: manifoldWorkerUrl },
];

// Pass to BuildProvider
<BuildProvider buildId={id} kernelOptions={extendedModules}>
```

The `id` field accepts any string -- known kernel names get autocomplete, but arbitrary third-party IDs work without type errors.

No changes to `kernel.machine.ts` are needed -- it dynamically creates and initializes all workers from the injected config.

## Best Practices

### Result Pattern

Always return result objects using the helpers:

```typescript
import { createKernelSuccess, createKernelError } from '@taucad/kernels';

// Success
return createKernelSuccess({ data });

// Error
return createKernelError({
  message: 'Human-readable error',
  type: 'code' | 'kernel', // code = syntax error, kernel = runtime error
});
```

### Keep `canHandle` Fast

The `canHandle` method is called for every file to determine which kernel to use. Don't:

- Initialize heavy runtimes
- Parse entire files
- Make network requests

Do:

- Check file extensions
- Look for quick signature patterns

### Export Formats

Define supported export formats as a static property:

```typescript
protected static override readonly supportedExportFormats: ExportFormat[] = ['stl', 'glb', 'step'];
```

### Use glTF-Transform for Output

For consistent 3D output, use the `@gltf-transform/core` library:

```typescript
import { Document, NodeIO } from '@gltf-transform/core';

const document = new Document();
// Build scene...
const io = new NodeIO();
const glb = await io.writeBinary(document);
```

### File System Access

Use the provided `fileReader` for filesystem operations:

```typescript
// Read file content
const content = await this.readFile('path/to/file.txt', 'utf8');

// Check if file exists
const exists = await this.exists('path/to/file.txt');

// List directory
const files = await this.readdir('path/to/dir');
```

### Logging

Use the built-in logging methods:

```typescript
this.debug('Processing file', { operation: 'createGeometry' });
this.warn('Deprecated feature used');
this.error('Failed to parse', { data: { line: 42 } });
```

## Middleware

Middleware wraps kernel operations (geometry creation, parameter extraction, export) using an onion model. Each middleware can intercept, transform, or short-circuit these operations.

### Architecture

Middleware is **dynamically loaded** in the Web Worker via `import(url)`. The main thread injects a `MiddlewareRegistrations` array — an ordered list of middleware entries — through the same dependency injection pattern used for kernel workers.

```
BuildProvider → build.machine → cad.machine → kernel.machine → Worker
                                                    ↓
                                            MiddlewareRegistrations[]
                                                    ↓
                                          import(url) per entry
```

### Creating a Middleware

Create your middleware in `apps/ui/app/components/geometry/kernel/middleware/`:

```typescript
import { z } from 'zod';
import { defineMiddleware } from '#components/geometry/kernel/middleware/kernel-middleware.js';

export const myMiddleware = defineMiddleware({
  name: 'MyMiddleware',
  version: '1.0.0',

  // Optional: Zod schema for configurable options (with defaults)
  configSchema: z.object({
    threshold: z.number().default(0.5),
  }),

  // Optional: Zod schema for internal per-operation state
  stateSchema: z.object({
    cache: z.map(z.string(), z.unknown()).default(new Map()),
  }),

  // Wrap hooks (all optional)
  async wrapCreateGeometry(input, handler, { logger, config }) {
    logger.debug(`Processing with threshold: ${config.threshold}`);
    const result = await handler(input);
    return result;
  },
});
```

### configSchema vs stateSchema

- **`configSchema`** — External settings from the main thread. Validated and merged with defaults on load. Reconfigurable at runtime via `configureMiddleware`.
- **`stateSchema`** — Internal per-operation state managed by the middleware itself. Not exposed to the main thread.

### Registering Middleware

#### First-party middleware

Add a `?url` import and entry to `defaultMiddlewareRegistrations` in `apps/ui/app/constants/kernel.constants.ts`:

```typescript
import myMiddlewareUrl from '#components/geometry/kernel/middleware/my.middleware.js?url';

export const defaultMiddlewareRegistrations: MiddlewareRegistrations = [
  // ... existing entries (order = execution order)
  { url: myMiddlewareUrl },
];
```

#### Third-party middleware

External consumers extend or replace the default config:

```typescript
import { defaultMiddlewareRegistrations } from '#constants/kernel.constants.js';
import type { MiddlewareRegistrations } from '@taucad/kernels';

const extendedEntries: MiddlewareRegistrations = [
  ...defaultMiddlewareRegistrations,
  { url: '/path/to/custom-middleware.js', config: { threshold: 0.8 } },
];

<BuildProvider buildId={id} middlewareConfig={extendedConfig}>
```

### Runtime Reconfiguration

Middleware can be reconfigured at runtime without restarting workers:

```typescript
buildRef.send({
  type: 'configureMiddleware',
  middlewareConfig: [
    { url: edgeDetectionUrl, enabled: false },
    { url: cacheUrl, config: { maxEntries: 50 } },
  ],
});
```

This flows through the full machine hierarchy and calls `configureMiddleware` on each active worker, which re-resolves config schemas with new values.

### Enabled Flag

Every middleware has a framework-level `enabled` flag that defaults to `true`. The framework automatically skips hooks for disabled middleware -- middleware authors never need to check it.

Resolution order: `MiddlewareEntry.enabled` > `defineMiddleware({ enabled })` > `true`

```typescript
// Middleware enabled by default (standard)
export const myMiddleware = defineMiddleware({
  name: 'MyMiddleware',
  // enabled defaults to true
});

// Middleware disabled by default (opt-in)
export const debugMiddleware = defineMiddleware({
  name: 'DebugMiddleware',
  enabled: false,
});

// Override at registration or runtime
{ url: myMiddlewareUrl, enabled: false }   // disable at registration
{ url: debugMiddlewareUrl, enabled: true } // enable at registration
```

Disabled middleware modules are still imported (so re-enabling via `configureMiddleware` is instant), but their hooks are skipped in the execution chain.
