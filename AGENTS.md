## Architecture

Tau is the AI-native CAD platform for the web (`tau.new`), built as an Nx monorepo with pnpm workspaces.

- **Frontend**: React Router v7, React 19, TypeScript, Tailwind CSS, Fumadocs
- **Backend**: NestJS API with Fastify, PostgreSQL (Drizzle ORM), Redis, Better Auth
- **CAD Engine**: Multi-kernel runtime (Replicad, JSCAD, Manifold, OpenSCAD, KCL)
- **AI**: LangGraph agent with tool-use (OpenAI, Anthropic, Vertex AI, Ollama, Together AI, Cerebras)

## Conventions

- Early returns to reduce nesting
- Composition over inheritance; functional programming patterns preferred
- Const declarations over function declarations
- `cn()`/`clsx` for conditional classNames, not ternary
- Max 3 parameters per function; bundle extras into an options object
- Vitest for tests; jsdom env for UI, node env for API
- Hybrid oxlint + ESLint linting; formatting via oxfmt (`.oxfmtrc.json`), not ESLint
- PostgreSQL with Drizzle ORM; schema in `apps/api/app/database/`; auth tables via Better Auth
- Investigate dependency source via `repos/` (managed by `repos.yaml` and `pnpm repos`), not `node_modules`. Use the `repos` skill to clone, add, or explore repos.

## Learned User Preferences

- Write failing tests first (TDD), then fix to pass; preserve existing tests; make minimal, targeted changes and run typecheck before considering done; remove tests for non-shipped APIs/behaviors rather than skipping them; do not mock Radix `Collapsible`/portal-rendered components in jsdom ŌĆö find and fix the real reason the test sees the wrong initial state instead of mocking around it
- State machines own lifecycle and state logic; UI clients send events only and never decide open/close; avoid ref/state for sync guards
- Follow policy docs when applicable: testing-policy, library-api-policy, xstate-policy, lint-policy, react-testing-policy, filesystem-policy, commit-policy, typescript-policy, agents-md-policy, context-engineering-policy, filesystem-context-policy, jsdoc-policy, documentation-policy
- Pin GitHub/dependency versions to exact commit hashes for reproducibility; use `pnpm patch` for dependency patches, do not manually create patch files; when behavior regressed from something that previously worked, prefer config changes over code changes ŌĆö find the regression; replicad source/replicad-opencascadejs edits in `repos/replicad` are in scope (close collaboration with upstream maintainer); fork remotes track upstream's primary branch ŌĆö assimp uses `master` (not main), tinyusdz uses `release` (not main); push fork commits to the matching upstream branch name; never hand-write full SHAs from short forms ŌĆö always run `git rev-parse <short>` to obtain the canonical full hash before pinning in CMakeLists/submodule configs (synthesised back-halves silently produce non-existent commits that break `FetchContent_Declare` at configure time); commit messages **and source code** on upstream/external repos must describe the change itself ŌĆö never embed local research doc IDs (`R1`, `R5`, `R8`, etc.) in commit subjects, code comments, or python codegen rationale, since other contributors don't have access to those references (audit `bindings.py`/`generateBindings.py` for stray `R1`/`R6`-style cites when applying gap-analysis recommendations)
- Use `react-virtuoso` for virtualization, not `@tanstack/react-virtual`; follow patterns in `combobox-responsive.tsx`; when virtuoso size/measurement bugs surface (e.g. tiny chat-message bubbles for very long text), fix via the slicing approach (cap items measured per pass, hand off to the existing virtuoso scroller) rather than removing virtuoso ŌĆö never trade away virtualization for a perf hit on long content
- Never blow away the entire IndexedDB database ŌĆö user work is stored there
- Prefer algorithmic, code-level solutions over bundler config or Vite plugins; middleware and transforms must be provider-agnostic (no model/provider-specific branching) ŌĆö handle all providers generically; text normalization/sanitization AND malformed-payload healing (e.g. interrupted-stream tool inputs that fail strict schema validation) belong at the API layer (middleware/transforms/preprocess), never in UI components ŌĆö the UI already renders fine without healing and per-render coercion misses every part already persisted in the DB; optimize for 3rd-party consumer DX; CAD export should not hardcode a single interchange format (e.g. GLB-only) or embed `@taucad/converter` as an internal runtime dependency ŌĆö treat converters and third-party exporters (e.g. API-based format transformers) as injectable plugins with multiple providers allowed concurrently; agent-surface APIs (RPCs, tool inputs, system prompts) require explicit parameters with no silent `?? mainEntryFile`-style fallbacks so the LLM must commit to the file/target it intends; cross-cutting agent UI metadata (e.g. reasoning timing) namespaced under the generic `common` AI-SDK provider key ŌĆö not a per-product `taucad` key ŌĆö and typed end-to-end via `MyUIMessage` schemas (never `unknown` providerMetadata)
- Avoid type assertion escape hatches (`as never`, `as unknown as`, unnecessary `as const` on returns); fix underlying type issues instead; use `z.input` (not `z.infer`) for consumer-facing Zod schema types ŌĆö preserves `.default()` optionality so callers see which fields are truly required; JSDoc codeblocks use explicit language tags (`typescript`/`javascript`, not `ts`/`js`); `@public` tag gates compile-checking; `@example` tags require `<caption>` per JSDoc spec (non-empty, no redundant "example" word); examples must reflect actual consumer usage patterns, not synthetic isolated calls; public JSDoc required for `libs/` and `packages/` only ŌĆö apps are exempt
- When asked to explore or investigate, present findings and analysis first; do not jump to code changes until implementation is explicitly requested; dig for the concrete root cause (the smoking gun) ŌĆö targeted fixes only, not broad investigation plans; for perpetual-loading / stuck-spinner bugs, never route on file extension ŌĆö content-sniff (BOM + NUL heuristic on first 512 bytes, VS Code `detectEncodingFromBuffer` style) and surface every outcome through a typed discriminated union so the editor render gate covers every state; validate hypotheses directly in source code (e.g., read emscripten/embind internals), no assumptions allowed; assess whether issues stem from incorrect implementation or broken architecture before fixing; never band-aid ŌĆö fix at source rather than post-processing; verify changes actually work (visually for UI, at runtime for logic) before claiming success; prefer the most correct architectural fix over the simplest ŌĆö correctness is not negotiable; adversarially review proposed approaches before committing to implementation; question manual additions when automated code generation should handle it; code generators must remain generic at the C++ type level ŌĆö no domain-specific class patterns, use standard type heuristics (templates, typedefs, inheritance) instead; avoid regex-based assertions in tests bound for upstream PRs ŌĆö they are hacky and indefensible in review, prefer principled assertions on parsed structures; for cross-pipeline parity (e.g. replicad vs opencascade glTF mesh), assert exact byte-for-byte output via TDD before fixing the underlying cause; also prefer round-trip geometry tests (export ŌåÆ re-import ŌåÆ re-mesh) over byte-count `>0` checks for export pipelines; geometry/test-suite tools must have non-overlapping semantic intent ŌĆö the LLM should never be forced to choose between two tests that achieve the same thing (e.g. mesh-continuity must measure disconnected-mesh count, not raw mesh count, so adding parts to a unicolor model never spuriously fails the test); geometry tests must derive results purely from the glTF mesh data ŌĆö never rely on kernel-supplied metadata or glTF `extras` fields (e.g. `assemblyParts` must compute overlaps from geometry alone) so future kernels only need to emit geometry, not extra annotations; chat-tool descriptions follow Claude Code's "When NOT to use:" pattern alongside the primary purpose for context-engineering DX; when removing language that "encouraged the LLM to skip a test" because the geometry unit lacked a top-level export, push the LLM the opposite way ŌĆö instruct it to add a kernel-specific top-level export so the file becomes testable, never to drop the test; tool descriptions point the LLM at the canonical schema (e.g. `<test_requirements>` for `test_model`) as a single source of truth rather than re-asserting requirement/format guidance in prose ŌĆö tool-description text stays identical across kernels, with cached tool wrappers differing only via error-path routing (`getKernelConfig`); streaming UI wrappers (e.g. `ChatActivitySection` "ExploringŌĆ”" header) must be a function of the **stable identity** of the underlying run (the first aggregatable group), never of a per-render count like `run.groups.length` that flips as parts arrive ŌĆö wrap-by-count produces visible mount/unmount flicker mid-stream and loses local state (`userToggleState`, animation timing); once the wrapper is visible for a research run it must remain mounted until a non-foldable category breaks the run, with all child collapsibles signalled via a React context (e.g. `ActivityFoldContext.disableInnerFold`) to skip their own chrome rather than relying on emergent state; remove temporary investigation logs once a diagnosis pass concludes ŌĆö keep only the durable plumbing that surfaces real-world WASM/render/network errors clearly (e.g. relaxed-simd parse failures, OCJS init faults, FileManager worker errors with named fields rather than `undefined undefined undefined`)
- Use `generatePrefixedId(idPrefix.*)` from `@taucad/utils/id` for all ID generation ŌĆö not `crypto.randomUUID()`; prefixed IDs use nanoid with typed prefixes defined in `@taucad/types/constants` (e.g., `idPrefix.log` = `'log'`)
- When extracting domain logic into standalone packages, invert dependencies ŌĆö the domain package owns its types/schemas, consumer packages depend on it (not the reverse); derive computed values from canonical sources rather than duplicating data across schemas; for runtime/kernel manifests authored as Zod, convert to JSON Schema during worker-side startup discovery so Zod instances never cross `postMessage` to the main thread; no backwards compatibility or deprecation paths for unreleased/internal APIs ŌĆö roll forward with breaking changes (after a Netlify-style cutover, also strip all stale comments referring to the old approach so the codebase doesn't carry historical artifacts); do not add shims or adapters for consumers that don't exist yet; kernels must be fully self-contained (no shared cross-kernel schemas/types like `occt-shared-schemas` ŌĆö each kernel duplicates its schemas to remain independently extractable); kernels treat a `main()` that returns `undefined` (no return statement or explicit `return undefined`) as a successful no-op with an empty `issues` array ŌĆö never emit warning issues for missing returns (tests assert `issues.length === 0`, not "warning when undefined"); appearance/material props (color, opacity, metalness, roughness, density) live as flat top-level keys on shape return values ŌĆö never nested under an `appearance` key ŌĆö for parity with Zoo's per-shape `appearance()` and to enable future engineering queries (e.g. stress under load via density); `opacity` is preferred over `alpha` (more descriptive, aligned with PBR conventions); `defineKernel`/`defineTranscoder` author APIs accept Zod schemas only ŌĆö JSON Schema and defaults are resolved internally so authors don't repeat themselves; schema-driven runtime APIs must use generic inference end-to-end (kernel formats, transcoder edges, `client.export` options) ŌĆö no `Record<string, unknown>` escape hatches when a schema can drive the type; `RuntimeClient` API splits intent: `client.render()` is for autonomous-mode rendering only (UI panes, may be superseded by newer renders), while `client.export(format, ?input)` is for on-demand use (CLI, testing, headless export) ŌĆö the format-only overload uses default export options for the format and the no-input overload reuses the kernel's most-recent native handle (a runtime invariant after every successful render) for near-instant live-pane exports; `connect()` is manually called by the cad machine (not auto on construction); the transport layer owns SAB awareness so consumer code never branches on capability ŌĆö SAB is preferred when pools are present and any non-SAB fallback work is short-circuited; `terminate()` must resolve cleanly with no hanging promises
- Chat composer toolbar, model/kernel selector triggers, and other paired breakpoint UIs: apply the same visible/control changes to both desktop and mobile layouts (including `chat-textarea-desktop.tsx` / `chat-textarea-mobile.tsx` when split) when the feature exists on both

## General Guidelines for working with Nx

- For navigating/exploring the workspace, invoke the `nx-workspace` skill first - it has patterns for querying projects, targets, and dependencies
- When running tasks (for example build, lint, test, e2e, etc.), always prefer running the task through `nx` (i.e. `nx run`, `nx run-many`, `nx affected`) instead of using the underlying tooling directly
- Prefix nx commands with the workspace's package manager (e.g., `pnpm nx build`, `npm exec nx test`) - avoids using globally installed CLI
- You have access to the Nx MCP server and its tools, use them to help the user
- For Nx plugin best practices, check `node_modules/@nx/<plugin>/PLUGIN.md`. Not all plugins have this file - proceed without it if unavailable.
- NEVER guess CLI flags - always check nx_docs or `--help` first when unsure

## Scaffolding & Generators

- For scaffolding tasks (creating apps, libs, project structure, setup), ALWAYS invoke the `nx-generate` skill FIRST before exploring or calling MCP tools

## When to use nx_docs

- USE for: advanced config options, unfamiliar flags, migration guides, plugin configuration, edge cases
- DON'T USE for: basic generator syntax (`nx g @nx/react:app`), standard commands, things you already know
- The `nx-generate` skill handles generator discovery internally - don't call nx_docs just to look up generator syntax
